That is, Rubric provides indication of all major assessment points; however, it is impossible to cover all different marking possibilities within the Rubric, and you need to make sure that all the requirements are fully addressed when submitting your Ass 2 solution.

                                                                                                                                                                  4) The comments section provided at the end of the Rubric, might have “Please fix in Ass 2” comments. Make sure to address those in Ass 2 to avoid losing marks for the stated issues.  

                                                                                                                                                                  5) Here is a list of few common issues found in Ass 1 submissions:

                                                                                                                                                                  • Not using coinDelay and flipDelay parameters inside the GameEngineImpl.flip (…) implementation. Not only this is required as part of the core system execution, as per the interface comments, but you will not be able to display intermediate coin faces (while "flipping") because the GUI cannot catch up with all the flips performed inside the Model.
                                                                                                                                                                  • Not using a separate thread when establishing Controller -> Model communication e.g. when calling model.flip (…). This in turn “freezes” the GUI making it unusable. The need to use a separate thread was stated on the specification doc, and also discussed during our chat sessions, with a running example provided in this forum (see my post from Sep 11, 2017).
                                                                                                                                                                  • Covering multiple players in the GUI. This is really unnecessary because each GUI instance should handle one player only! This needs to be fixed in Ass 2. Note that some students lost marks for such implementation, whereas others didn’t. This was based on the impact of multi-player handling (which again is not required) on the usability of your GUI. For example, most students who had the multi-player functionality going created proper UI components such as JList etc. that would list all the current players, allow the user to select a player from the list, and then use this player as a “default” player when placing the bets etc. This is totally fine because the usability of the system is not really reduced in such scenario. On the other hand, some students did not provide any graphical way for selecting the player, with the "place bet" function requiring the user to manually enter player’s id in the textfield. This is not user-friendly and is very error-prone – clearly you cannot expect the user to remember id values for potentially hundreds of players in the system. In cases like this, you would receive mark deduction. Please keep in mind that no matter how you managed this multi-player scenario, your GUI must handle one player only and this needs to be fixed in Ass 2.
                                                                                                                                                                  • Not providing a UI based function for updating/changing player’s points balance. This specific marking criterion refers to Ass 1 spec document: User Interface Requirements - "9. Update player’s credit points balance at run-time." Essentially you were required to provide a dedicated "Update points" button etc. that allows the player to update  / enter new points balance e.g. when the original balance hits zero as was discussed during our first Collaborate chat session. You Controller would then call GameEngine.setPlayerPoints(...) method provided for this particular purpose.
                                                                                                                                                                  • Not providing basic input validation. For example, you need to make sure that the provided credit points value is numeric. Throwing run-time exceptions with no indication to the user as to what went wrong when invalid input was provided is not a good option.

                                                                                                                                                                  Please make sure that the above issues are fixed in Ass 2 (if any of them apply to your current solution).
