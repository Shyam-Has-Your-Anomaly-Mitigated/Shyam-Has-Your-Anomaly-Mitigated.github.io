                                                              Callback-based design and MVC ---------------------------------------------------- Hi all, As per request posted in the Exam Preparation forum, I am providing a brief summary of the callback design concept in relation to the MVC architecture. There are two main strategies for implementing the MVC architecture. The first strategy revolves around a simple concept of "data modification / retrieval", where the Controller either: asks the Model to "add/update" its data based on the user input coming from the View; or "pulls" the required data from the Model, and then asks the View to update itself accordingly. In terms of our Ass 1, this particular MVC implementation strategy would cover pretty much all data manipulation functions such as addXYZ / getXYZ / removeXYZ etc. The second strategy, on the other hand, covers the "push-based" interactions originated from the Model. In this implementation, the Model itself notifies the View /or Controller when its state has changed, and pushes required data to the View/Controller if necessary.  In terms of the decision as to which strategy should be used when implementing the MVC, it all comes down to basic system requirements. If we have a simple system where the model will always have only one client (View/Controller) and all the functions executed on the Model are short-lived, the pull-based implementation is advised due to its simplicity. However, if we have a distributed system where the Model resides on a dedicated server (e.g. Ass 2) with potentially large number of diverse clients accessing/modifying it (e.g. java application, mobile app, php based website etc.), the push-based approach is the only option because all Model changes will need to be broadcasted to all relevant clients. Also, even in simple systems, this push-based approach can be useful if we have long-lived Model processes that need to periodically issue updates to the View/Controller. This is relevant to our flip () function in Ass 1, which could potentially run for a very long period of time (try increasing our delay parameters by a hundredfold and set random number of flips in the range of 50-100 :)). To this end, our Model was issuing callbacks to push coin state data back to the View/Controller. Now, coming back to the actual callback entity. Conceptually it is related to one of the core OO design patterns the Observer pattern, which was designed to reduce dependencies between major system entities e.g. the Model and the View/Controller in our case. This pattern is described in great detail in one of our recommended texts, Design patterns: elements of reusable object-oriented software, Addison-Wesley, 1995, and is also covered by Caspar in his GUI-related lectures. You can also find plenty of relevant resources on the Internet. In Java, this pattern, can be implemented very elegantly and with minimal effort by simply utilising two dedicated API entities: https://docs.oracle.com/javase/7/docs/api/java/util/Observable.html https://docs.oracle.com/javase/7/docs/api/java/util/Observer.html That is, we can make our Model to extend the Observable class and then call notifyObservers() when the Model state changes (the Observer/s would be our View/Controller that will need to implement the Observer interface). However, in our assignments, we decided to implement our own callback facility. Our GECallback is essentially a custom implementation of the Observer pattern, where Callback represents a mechanism for enabling a push-based interactions between the Model and the View/Controller, thereby removing unnecessary dependencies between them. You might ask why do we need to have this custom implementation instead of simply utilising existing Observable / Observer entities in Java API? The main reason would be additional requirements and constraints introduced by the client-server networking architecture of Ass 2. The Observer pattern can be implemented very easily in Java when all system components reside in the same location (Ass 1). However, it can become very difficult and complex once the Observable and the Observer become physically separated. Moreover, a pure Java-based implementation of the push-based Model interactions will be almost useless when dealing with clients developed using different programming paradigms/languages. Whereas custom Callback implementation strategy can support easier integration of new clients without any need to change the Model itself. For example, we can always add AppCallback, WebCallback etc. entities to our system to deal with new client types without requiring any changes to the underlying Model implementation. I highly recommend for you to read up more on the Observer design pattern, and also complete the Advanced exercise from: Topic 6: Exercises. Hint: it will be very useful when it comes to your final examination. Mikhail
